# 🎬 HearBeat — проєктний гайд до Digital Future Hackathon

**Команда (ролі):** Product Designer · Web Developer · ML Engineer · *(4-й учасник — TBD, див. розділ «Розподіл ролей»)*  
**Ніша:** Aging & Longevity  
**Дати хакатону:** 11–12 липня 2026  
**Формат гайду:** структуровано за блоками методички хакатону (0–6), кіно-метафора збережена  
**Пов'язані артефакти:** `brainstorm/hearbeat_product-vision.md` (повне продуктове бачення), `hackaton/hackathon-blocks-reference-guide.md` (загальний довідник по блоках)

**Рішення на вікенд:** показуємо не «майбутню телефонну систему», а один перевірений AI-flow: **голосовий чек-ін → акустичний аналіз → порівняння з baseline → зрозумілий сімейний сигнал**. Цільовий канал після хакатону — встановлений застосунок у літнього родича, який виглядає як простий вхідний дзвінок і спілкується з нашим сервером; стороння телефонія лишається опційним каналом масштабування, а не обов'язковим першим кроком.

### Стратегія «повна модель + мінімальне демо»

Журі купує **цілісну картину продукту**, а не обсяг коду за вікенд. Тому розділяємо два шари:

| Шар | Що це | Пріоритет | Результат |
|---|---|---|---|
| **Презентація (vision)** | Повна продуктова модель: проблема, ICP, AI-core, roadmap, бізнес, «умний дзвінок» | **Обов'язково** | Пітчдек, OnePager, усний пітч 3 хв |
| **Демо (proof)** | Один робочий сценарій, що доводить AI-core | **Мінімум обов'язковий** | Публічний лінк + Loom |
| **Полірування демо** | Те, що додаємо, якщо встигаємо | **Опційно** | Live ML, live-запис, кращий UI |

**Правило:** на сцені говоримо про **цільовий продукт**, на екрані показуємо **доказ ядра**. Не вибачаємось за scope — показуємо roadmap і чесно кажемо, що вже працює сьогодні.

**Формула пітчу:** «Ось як HearBeat виглядає в повноті → ось що ми вже зібрали за вікенд → ось наступні кроки.»

## Блок 0. Підготовка — режисер і знімальна група

### Логлайн проєкту

**HearBeat** — сервіс турботливих голосових чек-інів для сімей на відстані. У літнього батька або матері стоїть максимально простий застосунок, який виглядає як звичний вхідний дзвінок від сім'ї: людина натискає «відповісти», коротко говорить із HearBeat, а сервер аналізує голос і зміст розмови. Доросла дитина бачить зрозумілий тренд самопочуття: усе звучить як зазвичай чи сьогодні краще подзвонити самому.

Дорослі діти українців за кордоном не можуть фізично бути поруч із літніми батьками в Україні. Вони дзвонять раз на кілька днів, чують «все нормально», але тривога не зникає — бо немає об'єктивної опори між дзвінками.

### Роль ШІ (без неї фільму немає)

**Двошаровий AI-core:**

1. **Акустичний шар (первинний):** аналіз темпу мовлення, довжини пауз, варіативності інтонації + порівняння з персональним baseline людини → `vitality_score` і статус.
2. **Семантичний шар (другорядний):** коротке summary для сім'ї на основі транскрипту — без клінічних заяв.

Прибери ШІ — залишиться звичайне «як справи?» без сигналу. **AI-first тест пройдено.**

### Чому саме зараз і саме ця команда

- **Наукова база:** мовні біомаркери активно досліджують для когнітивного спадання, втоми й emotional wellbeing. На хакатоні не обіцяємо медичну точність, а демонструємо персональний тренд і пояснюваний сигнал.
- **Валідований ринок:** Sensi.AI показує попит на AI-care monitoring, але це B2B для care-агенцій; HearBeat бере вужчий B2C-wedge для сімей на відстані.
- **Показові UX-провали:** ElliQ — дорогий і нав'язливий формат AI-компаньйона; HearBeat позиціонується як тихий міст до реального дзвінка сім'ї.
- **Вільна ніша на хакатоні:** Aging & Longevity не зайнята серед учасників.
- **Рідкість у команді:** справжній ML-інженер — на хакатоні більшість роблять LLM-обгортки.

**Що НЕ обіцяємо:** діагноз, клінічну валідність порогів, заміну лікаря або «детектор деменції». Обіцянка демо скромніша й сильніша: показати, що голосові ознаки можна перетворити на ранній сімейний сигнал.

### Принцип «двох двигунів» на цьому проєкті

| Двигун | Де працюємо | Для чого |
|---|---|---|
| **ШІ-чат (Claude)** | Планування, брифи, промпти, пітч, дебаг-формулювання | Дешево думати |
| **Білдер (Lovable)** | Збірка UI, dashboard, публікація | Палити кредити лише на генерацію |
| **Claude Code / Python** | ML-пайплайн, синтетичні дані, інтеграція | Пара-програміст для акустичного ядра |

**Правило:** «що будувати?» — у чаті → «як зібрати?» — у білдері.

### North-star метрики вікенду

| Метрика | Що означає для HearBeat |
|---|---|
| **time-to-first-feature** | Перший запис голосу → score на екрані |
| **time-to-public-link** | Живе посилання, яке можна кинути журі |

### Критерій успіху MVP

До кінця хакатону має бути один публічний сценарій, який проходить без пояснень розробника:

```text
Відкрити лінк → відповісти на чек-ін → побачити оновлений dashboard →
зрозуміти, чому система радить «варто подзвонити».
```

Якщо є цей сценарій, проєкт можна показувати журі навіть без реальної телефонії, auth, кількох профілів і production-grade ML.

## Блок 1. Логлайн — біль, за яким прийдуть

### Проблема одним реченням

Дорослий українець за кордоном інтуїтивно тривожиться за стан літнього батька/матері в Україні, але не має об'єктивного способу зрозуміти, чи справді щось змінюється — лише рідкі дзвінки «на дотик».

### JTBD (Jobs To Be Done)

Людина «наймає» продукт, щоб **не пропустити момент, коли мамі/тату стало гірше** — не для розваги, не для контролю, а для своєчасної дії.

### Конкретні болі

| Біль | Як виглядає в житті | Наслідок |
|---|---|---|
| **Тривога після дзвінка** | Розмова була, але спокою немає | Немає даних, лише інтуїція |
| **Пізнє виявлення змін** | Про погіршення дізнаються від сусідів або вже після кризи | Втрачається час для м'якої реакції |
| **Почуття провини** | «Я далеко, я мало дзвоню» | Емоційно болюча мотивація до оплати |
| **Складність для батьків** | Додатки, логіни, дрібний текст | Літня людина не буде користуватись |
| **Суб'єктивність** | У кожному дзвінку намагаються «вловити тон» | Немає історії і baseline |
| **Розрізнена сім'я** | Один брат знає одне, сестра інше | Немає спільного сигналу |

### ICP (Ideal Customer Profile)

| Поле | Деталь |
|---|---|
| **Хто платить** | Українець/українка 28–45 років, живе за кордоном (Польща, Німеччина, Чехія та ін.) |
| **Кого підтримує** | Літній батько/мати 65+ в Україні, часто живе сам/сама |
| **Ситуація-тригер** | Дзвонить батькам раз на кілька днів; розмова звучить «нормально», але тривожно |
| **JTBD** | Не пропустити момент погіршення стану |
| **Біль + наслідок** | Тривога, провина за відстань, дізнається про проблему пізно |
| **Поточний обхід** | Дзвонить частіше «про всяк випадок», просить сусідів приглядати |
| **Де знайти** | Діаспора-чати, Facebook-групи українців у ЄС, Telegram-спільноти |

### Сигнали реальної проблеми

- ✅ **Частота** — тривога після кожного дзвінка, регулярно
- ✅ **Гострота** — емоційно болюча тема, почуття провини й безпорадності
- ✅ **Workaround** — дзвонять частіше, просять сусідів — це доказ реальності болю

### Масштаб проблеми

Майже 6 мільйонів українців знайшли прихисток за кордоном. Мільйони дорослих дітей фізично не можуть відвідувати літніх батьків, які лишилися самі в Україні — масова, вимірювана і зрозуміла українському журі проблема.

### Валідація за методом The Mom Test (план на суботу-ранок)

Питаємо ВИКЛЮЧНО про минулий досвід:

- Коли востаннє ти тривожився(лась) за батьків після дзвінка — що саме викликало тривогу?
- Що ти робиш зараз, коли відчуваєш, що щось не так, але не можеш перевірити особисто?
- Чи траплялось, що про погіршення стану ти дізнавався(лась) не від них самих — і наскільки пізно?

**Ціль:** 3 короткі розмови з людьми з діаспори до старту білду.

### Що вважаємо підтвердженням / спростуванням

| Сигнал | Рішення |
|---|---|
| 2+ респонденти згадують тривогу після «нормального» дзвінка | Логлайн підтверджений |
| Є реальний workaround: частіші дзвінки, сусіди, родичі, чати | Біль не вигаданий |
| Ніхто не хоче аудіоаналіз, але всі хочуть короткий family-summary | У пітчі сильніше подати summary, ML лишити як пояснюваний back-end |
| Люди бояться «стеження» більше, ніж хочуть спокою | Додати consent-first UX як ключову частину демо |
| Респонденти кажуть «я просто дзвоню щодня, мені не треба» | Звузити ICP до тих, хто не може дзвонити часто або живе в іншому часовому поясі |

### HADI-петля валідації

| Крок | Зміст |
|---|---|
| **Гіпотеза** | Діти-українці за кордоном тривожаться через відсутність об'єктивних даних, а не через відсутність спілкування |
| **Дія** | 3 Mom-Test розмови + скан діаспора-спільнот |
| **Дані** | Те, що реально почуємо, а не те, на що сподіваємось |
| **Інсайт** | Підтверджуємо або уточнюємо логлайн до старту знімального дня |

### Корисні інструменти для блоку 1

| Інструмент | Навіщо для HearBeat |
|---|---|
| **Facebook / Telegram діаспора-чати** | Проксі-сигнали болю своїми словами |
| **Відгуки ElliQ, Sensi.AI** | Готовий список, що не працює в існуючих рішеннях |
| **Google Trends** | Перевірка інтересу до aging/care тем |

## Блок 2. Сценарій — одна сцена, що тримає весь фільм

### AI-first тест

Прибери ШІ з HearBeat — залишиться звичайний дзвінок «як справи?» без сигналу. **З ШІ:** акустичний аналіз + порівняння з персональною історією + summary для сім'ї — це і є продукт. Тест пройдено.

### Value proposition

Для українця за кордоном, який не може відвідувати літнього батька в Україні, **HearBeat** перетворює короткий голосовий чек-ін на об'єктивний тренд самопочуття — на відміну від рідких дзвінків «навпомацки» — і дає привід зателефонувати саме тоді, коли це справді потрібно.

**Формула:** для [дорослої дитини за кордоном] у ситуації [після дзвінка з батьками, коли тривожно, але «все нормально»] HearBeat [регулярно збирає голосовий чек-ін і аналізує зміни] — на відміну від [суб'єктивних дзвінків без історії] — дає [зрозумілий тренд і сигнал «коли варто подзвонити»].

### Безжальний скоуп — ОДНА головна сцена

> Якби журі дозволили показати лише ОДНУ дію — яка доводить, що ідея працює?

**Головна сцена (демо):** емульований вхідний дзвінок HearBeat → короткі голосові відповіді → система аналізує акустичні ознаки → родич бачить статус, тренд і summary.

**Цільовий продукт (vision):** встановлений застосунок у батьків імітує простий вхідний дзвінок → той самий пайплайн аналізу на сервері → dashboard для сім'ї.

**Що саме доводить сцена:** не те, що ми вже побудували телеком-продукт, а те, що AI-core створює нову цінність між сімейними дзвінками: з'являється тренд, пояснення і зрозуміла дія.

### Драбина інтерфейсів

| Рівень | Що це | Статус для HearBeat |
|---|---|---|
| 1 | Бот / чат | Fallback: Telegram voice |
| 2 | Лендинг + waitlist | Хакатон: маркетинговий блок |
| **3** | **Веб-застосунок з ключовою функцією** | **Демо: емуляція дзвінка + dashboard** |
| 4 | App-based дзвінок + серверна оркестрація | Phase 1 після хакатону |

**Обраний рівень для демо:** 3 (веб з емуляцією дзвінка + dashboard).  
**Цільовий продукт:** рівень 4 (простий застосунок як основний канал; реальна телефонія — опція для пізнішого масштабування).

### Scope lock — що ПОЗА скоупом хакатону

- Реальна телефонія (Vapi/Twilio) — не перший обов'язковий крок; можлива Phase 2 для сімей без застосунку
- **«Умний дзвінок»** — адаптивна бесіда, VAD, уточнюючі питання, family prompt, календар — **лише в vision і на слайдах**, не в демо
- Когнітивна розминка — Phase 3, лише в roadmap
- Auth / реєстрація — не потрібно (публічний лінк)
- Мультимовність інтерфейсу
- Кілька профілів родини
- Клінічно валідовані пороги
- Діагностичні заяви

### Vision: «умний дзвінок» (не в демо, але в презентації)

Це **цільовий UX розмови**, який показуємо на слайдах і в roadmap — не будуємо на хакатоні.

```text
Цільовий дзвінок (Phase 1–1.5):
  1. Система слухає фразу за фразою (STT + VAD / довга пауза = кінець репліки)
  2. Після кожної відповіді — короткий аналіз (акустика + зміст)
  3. Наступне питання не з фіксованого списку, а адаптивне / уточнююче
  4. Доросла дитина задає «family prompt»: інтереси, теми, що не чіпати, тон
  5. Опційно: календар (дні народження, візити) → м'яке нагадування в розмові
```

| Елемент | Навіщо | Чому не в демо |
|---|---|---|
| **VAD / пауза** | Природна розмова, без «натисни стоп» | Потрібен стабільний real-time audio у застосунку |
| **Адаптивні питання** | Не сухий опитник, а жива бесіда | Потрібен LLM-оркестратор + контекст сесії |
| **Family prompt** | Персоналізація під конкретну сім'ю | Потрібен onboarding і збереження профілю |
| **Календар** | «Завтра день народження онука» | Інтеграція + налаштування — окремий спринт |

**На слайді 2–3:** показати схему «фіксований чек-ін (демо) → app-based дзвінок (Phase 1) → адаптивний дзвінок (Phase 1.5)». Журі бачить амбіцію; команда не тоне в scope.

### Когнітивна розминка (Phase 3, не в демо)

Після того як сім'я звикла до регулярних чек-інів, HearBeat може додати короткі голосові сесії на 5 хвилин: легкі вправи на пам'ять і увагу, розмови про спогади, питання за інтересами людини. AI підлаштовує складність і теми під конкретного користувача, а родич отримує короткий звіт про активність. Це не окремий «AI-друг», а природне розширення звичного дзвінка — коли вже є довіра до сервісу.

### Демо: три рівні готовності

| Рівень | Що входить | Коли зупиняємось |
|---|---|---|
| **MVP (must)** | Емуляція дзвінка → 3 фіксовані питання → score + dashboard з історією | Якщо часу мало — цього достатньо для пітчу |
| **Strong (should)** | + live-запис або TTS «втомлений» профіль → реальний ML-pipeline | Ціль команди на вікенд |
| **Polish (nice)** | + LLM-summary, кращий UI, waitlist, другий демо-сценарій | Тільки якщо MVP уже стабільний |

**Не розширюємо демо в бік «умного дзвонка»** — навіть якщо з'явиться час. Краще полірувати dashboard, дані й Loom, ніж робити напів-адаптивний чат, який ламається на сцені.

### Рішення першого білду

| Питання | Рішення |
|---|---|
| Скільки екранів? | 2: check-in для батька/матері + dashboard для родича |

## Блок 3. Реквізит — дані, що оживляють кадр

### Скільки даних потрібно

**20–50 реалістичних записів** — щоб dashboard не виглядав порожнім і демо-сценарій проходив від початку до кінця.

### Джерело даних для демо

**Синтетика через TTS (ElevenLabs або системний):** 20–30 голосових записів з навмисною варіацією темпу, пауз і інтонації — частина «бадьорих», частина «втомлених». Це демо-реквізит, який доводить, що тренд рахується коректно.

### Схема даних (Supabase, таблиця `checkins`)

| Поле | Тип | Опис |
|---|---|---|
| `id` | UUID | ідентифікатор запису |
| `profile_id` | UUID/text | один демо-профіль літньої людини |
| `audio_url` | text | посилання на аудіофайл |
| `created_at` | timestamp | дата й час чек-іну |
| `scenario_label` | text | `baseline` / `normal_day` / `tired_day` для демо-контролю |
| `transcript` | text | текст відповіді, якщо є STT або синтетичний сценарій |
| `features_json` | jsonb | темп, паузи, pitch-варіативність, energy |
| `vitality_score` | number (0–100) | розрахований індекс «бадьорості» |
| `status` | text | `normal` / `check-in needed` |
| `acoustic_delta` | text | пояснення відхилення («паузи +40%») |
| `summary_for_family` | text | коротке summary для родича |

### Мінімальний демо-набір

| Набір | Кількість | Для чого |
|---|---:|---|
| Baseline-записи | 10–15 | Показати «звичайний» голос конкретної людини |
| Normal day | 5–8 | Зелений сценарій без тривоги |
| Tired day | 5–8 | Жовтий сценарій із повільнішим темпом і довшими паузами |
| Edge cases | 2–4 | Короткий запис, тиша, шум — щоб не зламати демо |

### Нормалізація даних

- `vitality_score` — завжди число
- `created_at` — завжди дата
- `status` — обмежений набір значень, не вільний текст
- Жодних «склеєних» полів

### Чек-лист якості даних

- [ ] Кожне поле має один тип
- [ ] Немає дублікатів і порожніх рядків
- [ ] Даних досить для повного демо-сценарію
- [ ] У публічному демо немає реальних персональних даних (лише синтетика)
- [ ] Є мінімум 2 контрастні профілі: «бадьорий» і «втомлений» день

| Інструмент | Навіщо |
|---|---|
| **ElevenLabs TTS** | Синтетичні демо-голоси з варіацією темпу |

## Блок 4. Знімальний день — від брифу до робочого MVP

### 4.1. Бриф фічі (готовий для вставки в білдер)

**Структура:** контекст + користувач → ціль екрана → ключова дія → AI-роль → дані/вхід → обмеження → Definition of Done

- **Контекст + користувач (JTBD):** доросла дитина-українець за кордоном хоче за 30 секунд зрозуміти, чи все гаразд з батьками, не маючи змоги приїхати особисто
- **Ціль екранів:** (1) емульований дзвінок / голосовий чек-ін для батьків, (2) dashboard тренду для родича
- **Ключова дія:** батьки відповідають голосом на 3 короткі питання → родич бачить статус, тренд і summary
- **AI-роль:** акустичний аналіз голосу + порівняння з baseline + коротке summary; система визначає відхилення і формує зрозумілий статус
- **Дані/вхід:** аудіозапис (live або TTS) + 20–30 синтетичних «історичних» записів
- **Обмеження (чого НЕ робити):**
  - без реєстрації / логіну
  - без реальної телефонії на хакатоні — емуляція дзвінка
  - без діагностичних заяв — лише «звучить як зазвичай» / «варто подзвонити»
  - без мультимовності інтерфейсу
  - один батько — один родич
- **Definition of Done:**
  - [ ] екран «вхідний дзвінок» або голосовий чек-ін працює без логіну
  - [ ] користувач бачить consent-пояснення простими словами перед записом
  - [ ] запис або обраний TTS-зразок потрапляє в Supabase
  - [ ] ML-пайплайн повертає `vitality_score` і `status`
  - [ ] dashboard показує статус + графік тренду + summary
  - [ ] один «тривожний» сценарій проходить наживо
  - [ ] публічний лінк відкривається з чужого пристрою без логіну

| База даних | **Supabase** (+ RLS) | Історія check-in |
| Акустичний аналіз | **librosa / openSMILE** | AI-core, живий |

| *Беклог: телефонія* | *Vapi / Twilio / Bland* | *Опційний канал масштабування* |

### 4.5. Архітектура демо (спрощена)

```text
[Екран «вхідний дзвінок»]
        ↓
[Запис голосу / TTS-файл]
        ↓
[ML: librosa → vitality_score + status]
        ↓
[Supabase: checkins]
        ↓
[Dashboard: тренд + summary + «Подзвонити»]
```

**На хакатоні емітуємо канал дзвінка, але показуємо живий AI-flow.**

### 4.6. Fallback-план для ризикових інтеграцій

| Якщо ламається | План Б | Що чесно кажемо журі |
|---|---|---|
| Web Audio API в браузері | Кнопка «використати демо-відповідь» з TTS-аудіо | Канал запису не є core; core — аналіз голосу |
| Python ML API не встигає підключитись | Precomputed `features_json` і score для демо-записів | Пайплайн уже описаний; інтеграція — наступний технічний крок |
| Supabase оновлення не live | Dashboard читає останній запис після refresh | Демо все одно показує наскрізний шлях даних |
| LLM-summary нестабільний | Шаблонне summary на основі `scenario_label` і transcript | Summary — другий шар, не головний AI-core |

### 4.7. Цикл дублів

- Одна зміна — один дубль, тестувати після кожного
- Застряг на третьому дублі поспіль → відкат до робочого стану
- Найризикованіший шов: **ML ↔ Supabase ↔ dashboard** — робити раніше, не в останню ніч

### 4.8. Дебаг (4 кроки)

1. Згодувати помилку + контекст
2. Пояснити простими словами, що зламалось
3. Отримати 2–3 варіанти фіксу
4. Виправити обраний — не переписувати сусідні файли

**Doom loop:** кожен фікс ламає щось інше → стоп, відкат до останнього робочого чекпойнта.

### 4.9. Версіонування та безпека

- Чекпойнти Lovable після кожного робочого стану
- Секрети (API-ключі) — лише в `.env`
- **RLS у Supabase — обов'язково** перед показом (голосові записи — чутливі дані)
- Прибрати тестові «двері» і адмінку з публічного лінку

### 5 слайдів: повна модель (не копія демо)

| Слайд | Заголовок | Головна думка | Що показати |
|---|---|---|---|
| 1 | Проблема | «Все нормально» не знімає тривогу | ICP, біль, масштаб діаспори |
| 2 | Рішення + vision | Повний продукт: чек-ін → аналіз → сигнал | Схема для родича і батька/матері |
| 3 | AI-core + «дзвінок через застосунок» | Акустика + baseline; майбутнє — адаптивна бесіда | Два шари AI + блок vision (app-call, VAD, prompt, календар) |
| 4 | Ринок і бізнес | Діаспора, freemium, хто платить | Монетизація без медичних claims |
| 5 | Roadmap | Що в демо сьогодні vs Phase 1–4 | Чітко: **демо = ядро**, не весь продукт |

**Слайд 3 — найважливіший для вашої стратегії:** зліва — те, що працює в демо (акустика, baseline, dashboard); справа — «дзвінок через застосунок» як Phase 1 і «умний дзвінок» як Phase 1.5 / backlog.

### Roadmap для слайда 5

| Фаза | Що будуємо | Що показуємо журі |
|---|---|---|
| **Phase 0 (хакатон)** | Демо: емуляція + ML + dashboard | **Live proof** — ядро AI-flow |
| **Phase 1 (0–3 міс)** | Простий застосунок для батьків, що імітує вхідний дзвінок; пілот 10–20 сімей | Цільовий канал без складної телеком-інтеграції |
| **Phase 1.5** | «Умний дзвінок»: VAD, адаптивні питання, family prompt, календар | **Vision на слайді** — не в демо |
| **Phase 2 (опційно)** | Реальна телефонія для сімей без застосунку або з кнопковими телефонами | Канал масштабування, не умова першого запуску |
| **Phase 3 (3–6 міс)** | Когнітивна розминка | Розширення після довіри |
| **Phase 4 (6+ міс)** | Мультимовність, B2B2C, клінічна валідація | Масштаб |

### Що сказати на Q&A

**Про емуляцію дзвінка:**

> «Цільовий UX — не форма і не чат, а максимально знайомий сценарій вхідного дзвінка. На хакатоні ми емітуємо цей канал у вебі, а в першій реальній версії переносимо його в простий застосунок для батьків: голос так само аналізується, порівнюється з baseline і перетворюється на сигнал для сім'ї.»

**Про застосунок замість сторонньої телефонії:**

> «Для першої версії нам не обов'язково інтегруватися з Twilio або Vapi. Ми можемо поставити літньому родичу дуже простий застосунок, який виглядає як вхідний дзвінок: велика кнопка "відповісти", голосова розмова з нашим сервером, потім аналіз і dashboard для сім'ї. Це зберігає звичний сценарій для батьків, але різко спрощує технічний запуск і контроль якості. Справжня телефонія може з'явитися пізніше як додатковий канал.»

**Про «умний дзвінок» і фіксовані 3 питання в демо:**

> «У цільовому продукті розмова адаптивна: система слухає фразу за фразою, розуміє паузу, ставить уточнюючі питання, а дитина задає контекст через family prompt — інтереси, дати з календаря. Це Phase 1.5. На демо ми свідомо показуємо лише ядро — акустичний аналіз і сімейний сигнал — щоб не розмазати scope і не зламати live на сцені.»

## Ключові ризики й контраргументи

| Ризик | Контраргумент |
|---|---|
| «Це шум, не сигнал» | Не діагноз; персональний baseline; наукова база |
| «Батьки не відкриють сайт» | Цільовий UX — встановлений застосунок з інтерфейсом дзвінка; веб — лише демо |
| «Схоже на стеження» | Прозорість: батьки знають про запис і мету |
| «False positive буде тривожити сім'ю» | Не пушимо діагноз; даємо м'яку дію «краще подзвонити», а не паніку |
| «Просто LLM-обгортка» | Акустичний шар — первинний AI-core |
| «Як ElliQ» | Не «друг», а сигнал для реальної сім'ї |
| «Демо без справжнього дзвінка» | Емітуємо канал, не аналіз; перший реальний канал — app-based дзвінок, телефонія опційна |
| «Live ML не встигне» | Є fallback із precomputed features; пітч не залежить від магії live API |
| «Демо простіше за слайди» | Свідомий вибір: продаємо повну модель, доводимо ядро; roadmap пояснює різницю |
| «Чому не адаптивні питання вже зараз?» | Потрібні стабільний real-time audio, VAD і оркестратор; на хакатоні — фіксований чек-ін як proof of AI-core |
