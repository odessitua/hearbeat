# 📖 Довідник по блоках хакатон-методички + корисні інструменти

**Призначення файлу:** загальний довідник — що означає кожен блок гайду хакатону, навіщо він потрібен, і які інструменти (включно з тими, яких немає в оригінальному гайді) можуть бути корисні саме нашій команді (Product Design + Web Dev + ML).

Не прив'язано до конкретної фічі — можна використовувати як шпаргалку в будь-який момент вікенду.

---

## Блок 0. Підготовка — режисер і знімальна група

### Що це означає
Базовий mindset-блок перед стартом. Головна ідея: ти керуєш процесом (режисер), а ШІ виконує (знімальна група) — оператор, монтажер, художник. Ти не пишеш код руками, а ставиш задачі, перевіряєш результат і ухвалюєш рішення.

**Ключовий принцип "двох двигунів":** думай в одному інструменті, будуй в іншому.
- Чат-модель (Claude/ChatGPT) — дешево/безкоштовно, тут плануєш, пишеш брифи, ловиш баги
- Білдер (Lovable) — палить кредити за кожне повідомлення, тут збираєш сам продукт

**Дві метрики, на які орієнтуватись:** time-to-first-feature (як швидко з'явилась перша робоча фіча) і time-to-public-link (як швидко є жива публічна версія). Не обсяг, а швидкість до живого.

### Корисні інструменти (крім Claude + Lovable з гайду)
| Інструмент | Навіщо |
|---|---|
| **Claude Code** (десктоп) | Якщо потрібен більш глибокий контроль над кодом, ніж дає Lovable — працює як пара-програміст |
| **Cursor** | Альтернатива Claude Code для тих, хто звик до IDE-подібного інтерфейсу |
| **Notion / Linear (швидкий таск-трекер)** | Тримати список "що зробили / що в беклозі" видимим для всієї команди — особливо корисно з 3-4 людьми |
| **Excalidraw** | Швидкі скетчі архітектури "на серветці" перед тим, як писати бриф |

## Блок 1. Логлайн — біль, за яким прийдуть

### Що це означає
Пошук і формулювання реальної проблеми користувача ДО того, як почати будувати рішення. Найдорожча помилка хакатонів — "немає реальної потреби в ринку" (≈42% причин провалу стартапів за аналізом CB Insights).

**Ключові інструменти методу:**
- **JTBD (Jobs To Be Done)** — людина не "купує продукт", вона "наймає" його на конкретну роботу в своєму житті
- **The Mom Test** — питай про минулу поведінку ("коли востаннє ти..."), а не гіпотетику ("чи хотів би ти...") — так навіть мама не зможе тобі збрехати з ввічливості
- **HADI-петля** — гіпотеза → дія → дані → інсайт, швидкий цикл перевірки за короткий час
- **ICP (Ideal Customer Profile)** — детальний портрет: хто, ситуація-тригер, JTBD, біль+наслідок, поточний обхід, де знайти

**Сигнали реальної проблеми:** частота (болить регулярно) + гострота (втрачають час/гроші/нерви) + workaround (люди вже кустарно щось роблять, аби обійти проблему).

### Корисні інструменти
| Інструмент | Навіщо |
|---|---|
| **Reddit / Facebook-групи / Telegram-чати профільної теми** | Проксі-сигнали болю — люди вже описують проблему своїми словами, без вашого запиту |
| **Google Trends** | Швидко перевірити, чи взагалі шукають цю проблему |
| **AnswerThePublic (безкоштовний ліміт)** | Побачити реальні формулювання запитів людей навколо теми |
| **App Store / Google Play відгуки конкурентів** | Готовий список болів — люди самі перелічують, що їх бісить у існуючих рішеннях |

## Блок 2. Сценарій — одна сцена, що тримає весь фільм

### Що це означає
Вирізання скоупу до ОДНІЄЇ ключової фічі, яку реально довести до робочого стану за вікенд. Головний ворог хакатону — розповзання скоупу ("а давай ще додамо...").

**AI-first тест:** приберіть ШІ з продукту — якщо він ще має сенс, це не AI-first, а "апка з кнопкою ШІ для галочки".

**Драбина інтерфейсів** (бери найнижчий рівень, що доводить цінність):
1. Бот/чат
2. Лендинг + одна дія/форма лідів
3. Веб-застосунок з однією ключовою функцією
4. Мобільний / складна логіка + БД (майже завжди поза скоупом хакатону)

**Value proposition формула:** для [хто] у ситуації [коли] продукт [що робить] — на відміну від [статус-кво] — дає [цінність].

### Корисні інструменти
| Інструмент | Навіщо |
|---|---|
| **Miro / FigJam** | Візуальний мапінг скоупу "що входить / що в беклог" — особливо корисно для дизайнера в команді |
| **Figma (швидкі wireframes)** | Якщо потрібно узгодити структуру екранів ДО того, як віддати бриф у білдер — економить дублі |

## Блок 3. Реквізит — дані, що оживляють кадр

### Що це означає
Демо без даних виглядає мертвим. Потрібно 20-50 реалістичних записів, щоб інтерфейс "ожив" і демо-сценарій пройшов від початку до кінця. Це не продакшн-база, а реквізит.

**Джерела даних:** синтетика від ШІ (найшвидше), публічні набори (Kaggle), експорт із наявного (Sheets/CSV).

**Нормалізація:** кожне поле — свій тип (число, дата, текст), жодних "склеєних" полів.

### Корисні інструменти
| Інструмент | Навіщо |
|---|---|
| **Faker.js / Python Faker** | Генерація реалістичних синтетичних даних (імена, дати, тексти) програмно, швидше за ручний промпт-запит |
| **Mockaroo** | Веб-генератор тестових датасетів із гнучкими полями, безкоштовний ліміт |
| **Kaggle Datasets** | Реальні відкриті дані, якщо потрібна справжня фактура, а не синтетика |
| **ElevenLabs / інший TTS** | Якщо потрібні синтетичні аудіо/голосові дані для демо |

## Блок 4. Знімальний день — від брифу до робочого MVP

Найдовший блок, розбитий на 6 підетапів.

### 4.1 Бриф — сценарій для ШІ
**Що означає:** хороший бриф не пишеться одним промптом, а збирається через інтерв'ю — ти змушуєш ШІ спершу тебе допитати, а вже потім писати специфікацію.

**Структура брифу:** контекст+користувач (JTBD) → ціль екрана → ключова дія → AI-роль → дані/вхід → обмеження → Definition of Done.

### 4.2 Вибір білдера — дерево рішень
- Конструктор із вбудованим деплоєм (Lovable, Bubble, Webflow, Framer) — дефолт хакатону
- Самостійно з ШІ + хостинг (GitHub → Vercel/Netlify) — більше контролю
- No-code без деплою (Airtable, n8n, Make) — для автоматизацій

**Правило:** один основний білдер, не два паралельно.

### 4.3 Цикл дублів
Промпт → генерація → запуск → перевірка → правка. Одна зміна — один дубль. Застряг на третьому дублі поспіль — не латай, відкотись.

### 4.4 Логіка, дані та інтеграції
Auth майже ніколи не потрібен на цьому хакатоні — ТЗ вимагає публічний лінк без реєстрації. Секрети — лише в `.env`.

### 4.5 Дебаг
Цикл за 4 кроки: згодуй помилку → поясни простими словами → 2-3 варіанти фіксу → виправ обраний. **Doom loop** — коли кожен фікс ламає щось інше — сигнал відкотитись до робочого стану, не латати далі.

### 4.6 Версіонування й безпека
Чекпойнти/коміти після кожного робочого стану. RLS (Row Level Security) у Supabase — увімкнути перед показом, якщо є будь-які дані користувачів.

### Корисні інструменти (додатково до Lovable/Supabase/GitHub з гайду)
| Інструмент | Навіщо |
|---|---|
| **Claude Code** | Пара-програміст для складніших backend/ML частин, які Lovable не тягне нативно |
| **Postman / Insomnia** | Швидко перевірити API-виклики окремо від UI, коли щось не працює — ізолювати проблему |
| **Vercel / Netlify** | Якщо частина продукту (наприклад, ML-скрипт як API) виходить за межі Lovable |
| **Sentry (free tier)** | Опційно, якщо хочеться бачити реальні помилки під час демо в реальному часі |
| **librosa / openSMILE (Python)** | Для будь-яких задач з аудіо/голосовим аналізом — стандарт індустрії, безкоштовні |
| **Whisper (OpenAI, є безкоштовні варіанти запуску)** | Транскрипція аудіо в текст, якщо потрібен текстовий шар аналізу |

## Блок 5. Прем'єра — лінк, який не соромно кинути журі

### Що це означає
Продукт стає реальним лише коли він публічний. "Працює у мене" ще не значить "працює для всіх" — обов'язково перевірити на чужому пристрої, іншому браузері, без свого логіну.

**Шляхи публікації:** Lovable Publish (дефолт, найшвидше) → свій код + Vercel/Netlify → no-code, що вже живе онлайн.

### Корисні інструменти
| Інструмент | Навіщо |
|---|---|
| **BrowserStack (free trial)** | Перевірити на реальних різних пристроях/браузерах без фізичного доступу до них |
| **Google PageSpeed Insights** | Швидка перевірка, чи не вантажиться демо занадто довго на слабкому інтернеті журі |

## Блок 6. Трейлер для журі — демо, що продає за 3 хвилини

### Що це означає
Гарний продукт із поганим демо програє посередньому з чітким трейлером. Журі купує історію, не код.

**Арка пітчу:** гачок → проблема+ICP → демо наживо → чому AI core value → метрика+бізнес → що далі.

**Три позиції журі:**
- А (бізнес/інвестор) — ринок, монетизація, чому зараз
- Б (продукт/тех) — чи справді AI-first, чи працює наживо
- В (користувач/impact) — чи біль реальний, чи є "вау"

**Обов'язкові артефакти:** пітчдек PDF ≤5 слайдів, OnePager JPG 2400×3500, публічне посилання на демо.

**Бізнес-модель за 30 хв:** монетизація, хто платить і чому, розмір ринку (груба оцінка), базова юніт-економіка (CAC vs LTV).

### Корисні інструменти
| Інструмент | Навіщо |
|---|---|
| **Canva** (є пряма інтеграція в цьому чаті) | Пітчдек і OnePager без ручної верстки — можна згенерувати каркас заздалегідь |
| **Gamma.app** | Альтернатива для швидкої збірки деки з тексту |
| **Loom** | Запасне відео "золотого шляху" — must, бо лайв ламається в найгірший момент |
| **Descript** | Якщо потрібно швидко змонтувати/почистити відео-демо (вирізати паузи, помилки) |

## Загальний глосарій ключових термінів (коротко)

| Термін | Значення |
|---|---|
| **Вайбкодинг** | Створення продукту зі ШІ природною мовою — ти описуєш, ШІ генерує код |
| **MVP** | Найпростіша робоча версія, що доносить одну ключову цінність |
| **AI-first** | ШІ в самій петлі цінності продукту, не прикручений збоку |
| **Scope lock** | Звичка явно фіксувати, що ПОЗА скоупом, у кожному промпті |
| **North-star метрика** | Головний орієнтир (у нас: time-to-first-feature, time-to-public-link) |
| **Doom loop** | Коли кожен фікс ламає щось інше — сигнал відкотитись, не латати |
| **RLS** | Row Level Security — правила бази, хто які рядки може бачити/редагувати |
| **Smoke-тест** | Швидка перевірка, що головний сценарій працює не лише в тебе |
